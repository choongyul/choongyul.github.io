블로그 만들며 배운 Submodule

September 21, 2020 in Git

들어가며

블로그를 만들기 위해 HUGO Quick-Start 가이드를 보는데

git submodule add https://github.com/budparr/gohugo-theme-ananke.git themes/ananke

이 보인다. ‘엥’ 이건 뭥미? Submodule이 무엇인지 살펴보자!

Submodule 이란 뭘까?

Submodule 이란, Dependency 관리 시스템이라 생각해도 좋다.

이렇게 가정해보자!

당신은 데이터 베이스를 사용하는 웹 애플리케이션을 제작해야 한다. 데이터베이스와 어떻게 연결해야 할 것인가? 당신은 연결을 위해 Maven 프로젝트를 만들고 필요한 Repository와 Dependency를 pom.xml에 추가하기로 마음먹었다. 근데 문제가 발생했다. 연결이 되지 않는다. 연결 라이브러리 소스코드를 수정해야 할 것 같다. 데이터베이스 업체에 문의했더니 수정은 알아서 하라며 Github 주소를 대뜸 던져 준다.

이 순간 필요한 것이 git submodule이다.

블로그를 만들면서 보았던 명령은

git submodule add https://github.com/budparr/gohugo-theme-ananke.git themes/ananke

Ananke 테마의 소스코드를 themes/ananke 폴더로 다운로드하라는 의미로 해석할 수 있다.

git clone과 git submodule 차이

근데 왜 Clone을 사용하지 않고 Submodule을 사용할까? Clone을 해도 사용에는 지장이 없다. 그래서 대부분의 테마 설치 가이드에는 Clone과 Submodule 방법을 동시에 기술해 놓는다.(하지만 대부분이 Submodule 사용을 권장하고 있다.) 단지, 둘의 차이는“Upstream”의 변경 사항에 대한 추적성에 있다. Clone을 하면 변경을 모두 잃어버린다.

반대로, Submodule의 사용은 Upstream의 변경을 계속 추적할 수 있게 해준다. 메인 프로젝트 레포지토리와 외부프로젝트 레포지토리를 분리하여 Update와 Commit을 가능하게 한다. 그렇기 때문에 Submodule은 외부 프로젝트 소스코드의 버전을 세심하게 관리해야 할 때 사용된다.

Submodule 사용 사례

  1. Submodule로 받아온 프로젝트를 특정 버전으로 고정시키고 싶을 때
  2. Submodule로 받아온 프로젝트의 변경을 추적하고 싶을 때
  3. 프로젝트의 일부를 제 3자에게 위임하고 특정 시점에 작업을 통합하려 할 때

HUGO에서 블로그를 테마를 관리할 때 Submodule 사용을 권장하는 이유도 위 3가지 사례와 동일하다.

  • 첫째, 테마를 특정 버전으로 고정시킬 수 있고
  • 둘째, Upstream의 변경 사항을 추적할 수 있다.
  • 마지막으로, Git 프로젝트가 이미 존재하는데 git clone을 하는 조금 이상해 보인다. 다운로드 한 테마 싱크를 맞추려면 수정이 필요하지 않을까?

테마를 다운로드 한 디렉터리로가서 .git 폴더를 삭제해야 한다. 그래야 git status, git update 명령을 정상적으로 사용할 수 있다.

마치며

소스코드 전체를 다른 프로젝트와 공유한다. Github 철학과 잘 맞는다. 브라우저 개발처럼 큰 프로젝트를 진행 할 때는 꼭 필요하지 않을까? 나도 꼭 한번 실무에서 사용할 수 있는 기회가 있으면 좋겠다. 각 모듈 별로 Repository를 분리해서 관리해야 할 정도로 큰 프로젝트에 투입될 기회 말이다.
